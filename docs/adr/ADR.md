## ADR-001: ターゲットプラットフォームを Pebble Emery に限定する

- **日付**: 2026-06-14（コミット履歴より再構成）
- **状態**: 採用
- **決定**: `appinfo.json` の `targetPlatforms` を `["emery"]` のみとする。
- **理由**: Emery はマイクを搭載した Pebble 最上位機種であり、Dictation API を使う本アプリには必須。他機種では `PBL_MICROPHONE` が未定義となり音声入力が動作しない。
- **トレードオフ**: Pebble Time 等の旧機種ユーザーは対象外になる。

## ADR-002: SDK バージョンを 2 から 3 に変更する

- **日付**: 2026-06-14（コミット `7469b26` より）
- **状態**: 採用
- **決定**: `appinfo.json` の `sdkVersion` を `"3"` に設定する。

## ADR-003: コンパニオン JS を ES5 で記述する

- **日付**: 2026-06-14（コミット `7469b26` より）
- **状態**: 採用
- **決定**: `src/pkjs/index.js` は ES5 構文のみ使用。アロー関数・`const`/`let`・`Array.prototype.findIndex` 等は不使用。
- **理由**: PebbleKit JS ランタイムは ES5 準拠であり、ES6+ 構文は実行時エラーになる。
- **トレードオフ**: コードが冗長になる（`firstNonSystemIndex()` 等の手動実装が必要）。

## ADR-004: AppMessage バッファサイズを 512 bytes に設定する

- **日付**: 2026-06-14（コミット履歴より再構成）
- **状態**: 採用
- **決定**: `app_message_open` の inbox/outbox サイズを `min(platform_max, 512)` とする。
- **理由**: Pebble のプラットフォーム上限が小さい環境でもクラッシュしないよう上限チェックを入れつつ、実用的なテキスト長を確保するために 512 bytes を選択。
- **トレードオフ**: 長い応答は JS 側で 500 文字にトリムされる（`text.substring(0, 500)`）。

## ADR-005: ウォッチ側履歴をリングバッファ 5 件とする

- **日付**: 2026-06-14（コミット `e36353f` より）
- **状態**: 採用
- **決定**: C 側の `HistEntry s_hist[5]` でウォッチ画面内の閲覧履歴を管理する。JS 側の会話コンテキスト（10 ターン）とは独立して管理する。
- **理由**: ウォッチのメモリ制約上、無制限に履歴を保持できない。Q: 128 B + A: 512 B = 640 B × 5 = 3.2 KB が現実的な上限。
- **トレードオフ**: ウォッチで閲覧できる過去 Q&A は最大 5 件のみ。JS 側の会話コンテキスト（10 ターン）より少ない。

## ADR-006: OpenAI モデルとして gpt-4o-mini を採用する

- **日付**: 2026-06-14（コミット履歴より再構成）
- **状態**: 採用
- **決定**: API リクエストのモデルを `gpt-4o-mini` に固定する。`max_tokens: 200`。
- **理由**: スマートウォッチの小画面では長文応答は不要。`gpt-4o-mini` は低レイテンシかつ低コストであり、腕時計の UX に適する。

## ADR-007: API キーの永続化に localStorage を使用する

- **日付**: 2026-06-14（コミット `5f639a7` より）
- **状態**: 採用（`5f639a7` で競合する保存場所を localStorage に一本化）
- **決定**: PebbleKit JS の `localStorage.setItem('openai_api_key', key)` のみを正とし、他の保存手段（`Pebble.getLocalStorageItem` 等）は使わない。
- **理由**: PebbleKit JS の `Pebble.getLocalStorageItem` と `localStorage` が別ストレージであることによる二重管理バグを解消するため。
- **トレードオフ**: `HARDCODED_API_KEY` 変数は開発テスト用にのみ残す（本番では空文字）。

- **日付**: 2026-06-14（コミット `e8b5812` より）
- **状態**: 採用

## ADR-009: 設定 UI を GitHub Pages でホストする

- **日付**: 2026-06-14（コミット履歴より再構成）
- **状態**: 採用
- **決定**: `config/index.html` を `https://kukai.github.io/pebble-wrist-agent/config/` で公開する。JS 側はこの URL を `?v=N` のキャッシュバスト付きでオープンする。
- **理由**: Pebble の `showConfiguration` はリモート URL を必要とし、アプリバンドル内の HTML を直接参照できない。GitHub Pages は無料かつ静的ファイルのホスティングに適する。
- **トレードオフ**: 設定 UI の更新はリポジトリへの push が必要。スマートフォンのキャッシュ対策として `?v=N` を手動インクリメントする運用が必要。

## ADR-011: CONFIRM 画面を廃止して音声認識後に即送信する

- **日付**: 2026-06-21
- **状態**: 採用
- **決定**: 音声認識成功時に CONFIRM 画面を経由せず、`send_query()` を直接呼び出す。`SCREEN_CONFIRM` および関連 UI・クリックハンドラをすべて削除。
- **理由**: LLM は多少の音声認識誤りを文脈から補完できるため、送信前確認の価値が低い。ステップ削減（ボタン2回→1回）による UX 改善を優先。
- **トレードオフ**: 誤認識テキストを確認・キャンセルする手段がなくなる。誤認識時は ANSWER 画面から HOME に戻って再質問する運用で対応。

## ADR-012: PR #3 後の「STT→AI リクエスト消失」の原因は UUID 変更による API キー消失（コード変更なし）

- **日付**: 2026-07-08
- **状態**: 採用（インシデント記録。ADR-011 の即時送信方式は実機検証済みとして維持）
- **事象**: PR #3 適用後、音声認識は成功するのに AI への問い合わせが行われなくなった。当初は dictation コールバック内からの即時 `send_query()` が音声セッション解体と競合する仮説（遅延送信 + ウォッチ側リトライで対処）を立てたが、実機での切り分けの結果、原因は API キーの消失（JS が `error:no_api_key` を返す経路）と確認された。即時送信自体は問題なく動作している。
- **原因**: PR #3 で `appinfo.json` の UUID をプレースホルダから CloudPebble の UUID に変更した。PebbleKit JS の localStorage はアプリ UUID ごとに分離されるため、旧 UUID 時代に保存した `openai_api_key` が参照できなくなった。スマホアプリの設定画面にはキーが保存済みのように見えていたが、これは表示キャッシュであり実体は失われていた。
- **決定**: ウォッチ側コードは変更しない（遅延送信・リトライは実機で不要と確認されたため導入を撤回）。UUID 変更を伴うリリース時は、設定 UI から API キーを再入力する運用とし、CLAUDE.md / SPECS.md に注意事項として明記する。
- **教訓**: 認識成功後にフローが止まった場合、`pebble logs` で `[WA] handleQuery` の有無を確認すれば「ウォッチ→スマホ送信の問題」か「API キー/OpenAI 側の問題」かを即座に切り分けられる。

## ADR-013: 天気ツール（function calling）を Open-Meteo で実装する

- **日付**: 2026-07-09
- **状態**: 採用
- **決定**: OpenAI function calling で `get_weather(location?, date?)` ツールを追加する。天気データは Open-Meteo（Geocoding API + Forecast API）を使用し、場所未指定時は `navigator.geolocation` の現在地を使う（appinfo の `capabilities` に `location` を追加）。ツール実行 → 再問い合わせは最大 2 ラウンドに制限。tool 往復メッセージ（assistant の `tool_calls` / role=`tool`）は会話履歴に永続化せず、履歴には user と最終 assistant 応答のみ保存する。system プロンプトには毎リクエスト今日の日付を注入する（「明日」「週末」等の相対日付を LLM が解決するため）。
- **理由**: Open-Meteo は API キー不要のため、既存の OpenAI キー以外に秘密情報の管理を増やさない（設定 UI の変更も不要）。tool メッセージを履歴に残すと `MAX_HISTORY` トリムで assistant(`tool_calls`)/tool のペアが分断され OpenAI API がリクエストを拒否するため、永続化しない設計とした。
- **トレードオフ**: ツール利用時はレイテンシが増える（OpenAI → 位置解決 → 天気 API → OpenAI。各段にタイムアウトあり、最悪 ~50 秒）。商用天気 API と比べ地点精度・詳細度は限定的。ツール定義の分だけ毎リクエストの入力トークンが増える。
- **補足**: タイマーツールは未実装。実現にはウォッチ側の Wakeup API 対応（新 AppMessage キー、C 側のタイマー管理・通知 UI）が必要なため、別 ADR として検討する。

## ADR-014: ホーム画面を MenuLayer ベースの 3 セクションダッシュボードに再設計する

- **日付**: 2026-07-18
- **状態**: 採用
- **決定**: アプリ起動直後の HOME を、単一の音声入力画面から MenuLayer の 3 セクション構成に変更する。S1 = 「話す」（固定 1 行・常に初期選択）、S2 = アクティブなタイマー/ストップウォッチ一覧（0 件時は非表示）、S3 = 固定メニューテーブル（現在は「会話履歴」のみ。`s_fixed_items[]` 配列への追記だけで行を拡張できる）。S2 の行操作は Pebble 標準の ActionMenu API を使い、独自サブメニューは実装しない。S2 の時間表示はカウンタを持たず、TickTimerService（SECOND_UNIT・HOME 表示中のみ購読）で `layer_mark_dirty` し描画時にタイムスタンプ差分から再計算する。
- **理由**: Pebble Time 2 のクイックローンチ枠が既に埋まっており、タイマー等の新機能は WristAgent 内に統合する必要がある。MenuLayer 化により将来の項目追加（天気/メモ/Home Assistant）に行数決め打ちなしで耐えられる。HOME 表示のたびに「話す」を初期選択に戻すことで、従来の「起動して Select 即発話」の体感を維持する。
- **トレードオフ**: UP/DOWN がメニュー移動に割り当てられるため、従来の「HOME UP 長押しで会話リセット」は廃止。代替として S3 に「会話履歴」項目を設け、リセットは ANSWER（履歴）画面の SELECT 長押し (700 ms) に移設した（ユーザー確認済み）。旧 HOME の履歴カウンタ表示（"[履歴: N件]"）は廃止。
- **補足**: 要件上の「🎤」絵文字は Pebble システムフォントで描画できないため、行タイトルはテキスト「話す」とした。

## ADR-015: タイマー/ストップウォッチを Wakeup + PersistentStorage + function calling で統合する

- **日付**: 2026-07-18
- **状態**: 採用
- **決定**:
  - OpenAI function calling に `set_timer(duration_seconds, label?)` / `start_stopwatch(label?)` を追加。JS はツール実行時に新 AppMessage キー（`KEY_TIMER_SET`=4, `KEY_TIMER_LABEL`=5, `KEY_STOPWATCH_START`=6）でウォッチへ送信し、**ウォッチの ack を待たず楽観的に tool 結果を即時返却**する（レイテンシ最小化。予約失敗はウォッチ側ステータス表示で通知）。
  - ウォッチ側は固定長 4 スロットでタイマー/SW を管理し、PersistentStorage（キー 100+n）に構造体ごと永続化。タイマーは Wakeup API（cookie=スロット番号, notify_if_missed=true）で満了通知し、予約が排他ウィンドウ等で失敗した場合は 5 秒ずらして最大 8 回リトライする（Pomodoro で確認済みの制約への対応）。最短 30 秒未満の指定は 30 秒に切り上げ。満了時はバイブ + スロット削除。
  - タイマー/SW を音声セットした直後の LLM 応答（KEY_RESPONSE）受信時は、ANSWER でなく HOME に戻る（`s_pending_home` フラグ）。応答自体は履歴に保存され後から閲覧できる（ユーザー確認済み。ストップウォッチは要件どおり、タイマーも同挙動に統一）。
  - ActionMenu 経由の操作（一時停止/再開/リセット/削除、SW の Start/Stop/Lap/Reset）はすべてローカル処理で完結し、LLM 往復を挟まない。タイマー一時停止は Wakeup キャンセル + 残り秒保存、再開は再スケジュール。
  - 起動時に実行中タイマーの `wakeup_query` を検証し、発火済みなら削除、予約消失なら残り時間で再スケジュールするサニタイズを行う。
- **理由**: ストップウォッチの Start/Stop/Lap は LLM 往復のレイテンシに耐えられないため、開始トリガーのみ音声・操作はローカルというハイブリッド設計とした。tool 結果の楽観的返却は、AppMessage の ack 待ち（NACK リトライ込みで最悪数秒）が LLM の第 2 ラウンドを遅らせるのを避けるため。tool 往復を履歴に永続化しない方針は ADR-013 を踏襲。
- **トレードオフ**: スロット満杯・Wakeup 予約失敗が LLM に伝わらないため、応答文（「セットしました」）と実際の状態が食い違う可能性がある（ウォッチのステータス行表示で補完）。Wakeup は 1 アプリ最大 8 件だが、スロット上限 4 のため実用上問題ない。スロット構造体の Persist 書き込みはスロット単位（≤ 256 B/キー制限内）。

## ADR-016: HOME の BACK ボタンを明示的にサブスクライブする

- **日付**: 2026-07-19
- **状態**: 採用
- **事象**: PR #5（ADR-014）適用後、実機（Pebble Time 2）で複数回「Listening→発話→送信」サイクルを繰り返すと、HOME 画面で BACK ボタンの短押しが反応しなくなるタイミングが報告された。その状態で BACK を長押しすると、Pebble OS のシステム仕様により強制的にアプリが終了しランチャーへ戻る（これはアプリコードで変更不可能な OS 標準動作であり、それ自体はバグではない）。
- **調査**: `menu_layer_set_click_config_onto_window()` は UP/DOWN（スクロール）と SELECT のみをバインドし、BACK は明示的に触れない。公式ドキュメント（[Clicks - Pebble Developers](https://developer.rebble.io/docs/c/User_Interface/Clicks/)）によれば、BACK は何もバインドしなければデフォルトで「ウィンドウを閉じる」動作にフォールバックするはずであり、通常は MenuLayer 使用時でも問題なく動作する。ただし実機での再現性・タイミング依存の性質から、暗黙のデフォルト挙動への依存を排除する方が安全と判断した。
- **決定**: HOME の Click Config Provider を `menu_layer_set_click_config_onto_window()` に一任せず、一度だけ `window_get_click_config_provider()` でその関数を保存し、`home_click_config_provider()` でラップして呼び出したうえで BACK を明示的に `window_single_click_subscribe()` する（[Pebble: How to override the back button when you are using a menu layer](https://gist.github.com/sarfata/10574031) の定石パターンを踏襲）。あわせて LOADING 画面も BACK を明示的に no-op へバインドし、リクエスト中に BACK でアプリが終了できてしまう抜け道（「操作無効」の仕様に反する）を塞いだ。
- **トレードオフ**: なし。既存の「BACK でアプリ終了」という挙動自体は変えておらず、暗黙のデフォルトに依存していた部分を明示化しただけ。根本原因（BACK が時々反応しなくなる具体的なメカニズム）は実機ログでの追加検証なしには完全には特定できていないため、再発した場合は `pebble logs` でクリックイベントの受信有無を確認すること。

## ADR-017: タイマー/ストップウォッチを HOME の常設メニュー（S3）からも追加できるようにする

- **日付**: 2026-07-19
- **状態**: 採用
- **決定**: HOME S3 に「タイマー」「ストップウォッチ」の固定項目を追加する。「タイマー」選択時、アクティブなタイマーが 1 件もなければタイマー設定画面（ActionMenu によるプリセット選択: 1/3/5/10/15/20/30分）へ自動遷移し、既にあれば該当タイマーの既存 ActionMenu（S2 行選択時と同じ）を開く。「ストップウォッチ」も同様の構造だが、未設定時はプリセット不要のためボタン操作のみで即座に開始する。既存タイマー/SW の ActionMenu にも、空きスロットがある場合に限り「新しいタイマー」/「新しいストップウォッチ」項目を追加し、既に1件動いていても追加できるようにする。
- **理由**: 音声を使わずボタン操作だけでタイマー/SW を開始・追加できる導線を求められたため。時間指定は Pebble 標準 API に数値入力ウィジェットが無く、既に実績のある ActionMenu を再利用する方が実装・検証コストが低いためプリセット選択方式とした（NumberWindow 相当の自作 UI は今回見送り）。プリセット選択・新規追加とも、音声経由の `set_timer`/`start_stopwatch` と同じ `handle_timer_set()`/`handle_stopwatch_start()` を再利用し、ロジックの二重管理を避けている。
- **トレードオフ**: 複数のタイマーが同時にアクティブな場合、S3「タイマー」は先頭の 1 件の ActionMenu しか開かない（2 件目以降は S2 から個別に操作する必要がある）。プリセット以外の任意の時間（例: 7分）はボタン操作のみでは指定できず、音声か最寄りのプリセットで代替する必要がある。

## ADR-018: 会話履歴をユーザーがリセットするまで PersistentStorage に永続化する

- **日付**: 2026-07-19
- **状態**: 採用
- **事象**: アプリを再起動すると会話履歴（ANSWER 画面で閲覧する Q&A）が消えるという報告があったが、調査の結果これは PR #5 による regression ではなく、`s_hist` リングバッファは元々一貫してメモリ上にのみ保持されており、永続化の実装自体がこれまで存在しなかったことが判明した（ADR-005 は表示件数の上限を定めたのみで、永続化の可否には触れていない）。
- **決定**: `s_hist`（`HistEntry s_hist[HIST_CAP]`）を PersistentStorage に永続化する。1 キー最大 256 bytes の制約があるため、各エントリの `q`（128 bytes）は 1 キー、`a`（512 bytes）は 256 bytes ずつ 2 キーに分割して保存し、件数は別途メタキーに保存する。`push_history()` のたびに全件を書き直し、`reset` コマンド（ウォッチ側 SELECT 長押し、または JS 側 `reset_ok`）で全キーを削除する。件数の上限は ADR-005 で決定済みの `HIST_CAP = 5` をそのまま踏襲し、変更しない。
- **理由**: 「ユーザーがリセットするまで履歴は保持されてよい」という要望と、既にタイマー/SW スロットで確立している PersistentStorage 永続化のパターンを踏襲するのが自然だったため。件数上限を変える意思決定は特になされていないため、既存の ADR-005 の方針（メモリ制約に基づく 5 件）をそのまま流用した。
- **トレードオフ**: `push_history()` のたびに複数キーへの書き込みが発生し、Wakeup サニタイズ等と合わせてフラッシュ書き込み頻度がやや増える（実用上は許容範囲）。JS 側の `conversationHistory`（会話コンテキスト、最大 10 ターン）は今回の対応範囲外であり、引き続きアプリ再起動で失われる（ウォッチ側の閲覧履歴とスマホ側の LLM コンテキストは ADR-005 のとおり独立管理のため）。
